기존 시스템은 그대로 두고, "정산 결과가 맞는지"를 원장 공개 없이도 검증/증빙하는 레이어

코어 모듈 (최소 3개)

1

Rule Pack

정산 룰의 "고정"

  • 계약/프로모션 조건을 룰(JSON/DSL)로 표준화
  • 룰 해시(버전)로 "이번 정산은 이 룰로 계산"을 고정
2

Confidential Compute

FHE16 적용 구간

  • 민감 필드(단가/수량/마진/특약)만 암호문 상태로 계산
  • 나머지 키는 해시/토큰화로 최소 노출
3

Proof Pack

분쟁/감사 대응 증빙

  • 룰 해시, 입력 커밋(해시), 실행 ID, 환경 다이제스트
  • (선택) 이의제기 시 최소 공개로 검증 가능
05 How it Works

"벤더가 연산"하는 현실 시나리오 (GTM 쉬운 구조)

! 핵심: 원문 데이터는 절대 우리에게 오지 않음

연산은 고객이 이미 신뢰하는 운영 벤더(SI/BPO/정산대행)가 수행하고,
우리는 "엔진/컨테이너/에이전트"만 공급

참여자

A

지급자
제조사/브랜드사

B

수령자
총판/대리점/리테일

V

운영 벤더
ERP/SI/BPO/정산대행

Us

FHE16 엔진
+ 로컬 암호화 에이전트

데이터 흐름

1

A/B 각 사에서 ERP/정산시스템에서 데이터 추출

2

로컬 에이전트가: 표준 스키마 변환 → 민감 필드 암호화(FHE) → 서명/커밋 생성

3

A/B는 암호화 패키지만 V에게 전달 (원문은 밖으로 안 나감)

4

V는 자신의 인프라에서 우리 컨테이너를 실행

5

출력: 정산 결과 + Proof Pack (감사/분쟁 대응)

06 Value Proposition

고객 가치 제안 (ROI 스토리) — "암호학"이 아니라 "돈/시간"

비용 절감

  • 과지급/누락 감소 (합의 빠름, 오류 줄어듦)
  • 분쟁 처리 비용 감소 (메일 왕복/재계산/표본감사)
  • 정산 인력 투입 시간 감소

시간 단축

  • 월/분기 정산 마감 리드타임 단축
  • "상대 원장 요구 → 거부 → 표본검증 → 재계산" 루프 감소

수익/현금흐름

  • 정산 지연 해소로 미지급/미수금 꼬임 완화
  • 채널 인센티브 집행 빨라져 채널 성과 관리 용이

"엑셀 계산을 자동화하는 게 아니라, '원장을 공유할 수 없어서 자동화가 안 되던 회사-회사 간 정산 합의'를
FHE16으로 가능하게 만들어 분쟁과 마감 시간을 줄입니다."
