04 Problem

기관 채택을 막는 프라이버시 부재

이슈 현황 리스크
거래 정보 노출 온체인 결제 시 상대방/금액/시점 영구 공개 M&A/투자 정보 누출
내부통제 충돌 자산운용사/보험사 내부통제 기준: "거래정보 최소 노출" 컴플라이언스 불가
시세조종 위험 대량 거래 사전 노출 프론트러닝 가능
글로벌 기관 요건 해외 기관투자자 프라이버시 요구 글로벌 자금 유치 불가
참고: 금융위 토큰증권 가이드라인 (2023.02) / BIS Report: Privacy in CBDC (2023)
결론: 규제 친화적 프라이버시 없이는 기관 채택 불가
05 Solution

아키텍처: ZK Pool + FHE Audit Log

평시 거래 정보 완전 비공개
감사 시 다자 승인으로 선택적 공개
규제 요건 "필요 시 접근" 충족
기술 배경: Zcash Shielded Protocol / HE Standardization (HomomorphicEncryption.org)
06 Integration

KSDC 연동: Payment Leg에만 적용

연동 포인트

  • 입금(Shield): 스테이블코인 → ZK Pool 진입 시 기관코드 해시 바인딩
  • 결제: Pool 내 비연결 전송 (Atomic Settlement 연동)
  • 출금(Unshield): Pool → 스테이블코인 전환, Freeze 체크
핵심: Delivery Leg 변경 없음 / 기존 청산 로직 유지 / Payment Leg에만 레이어 추가
07 Data Flow

데이터 흐름: 무엇을 숨기고 무엇을 공개하는가

데이터 평시 감사 시 보호 방식
거래 금액 비공개 공개 가능 Pedersen Commitment
거래 상대방 비공개 공개 가능 ZK Pool 비연결성
거래 시점 비공개 공개 가능 FHE 암호화 타임스탬프
기관코드 해시 존재 증명 원본 공개 SHA-256 해시 (PII 미포함)
Freeze/Blacklist 상태 실시간 체크 이력 조회 온체인 상태 플래그
개인정보 처리 설계: KYC ID는 기관코드 해시(SHA-256)로 저장. 원본 매핑은 각 기관이 오프체인 보관. 온체인에 PII 미저장.
08 Governance

키 관리 거버넌스: t-of-n 분산 키

복호 절차

  1. 감사/수사 요청 접수 (문서 기반)
  2. 각 키 홀더 승인 (t개 이상)
  3. 부분 키 결합 → 복호 키 생성
  4. 대상 TX만 선택적 복호
  5. 접근 로그 불변 저장
핵심: 단일 기관의 단독 복호 원천 차단 / 모든 복호 시도 로깅

10 Success Criteria

PoC 성공 기준 (KPI)

지표 목표 측정 방법
처리량 (TPS) ≥ 50 TPS 동시 100건 부하 테스트
ZK 증명 생성 ≤ 15초 Shield/Unshield 평균
ZK 증명 검증 ≤ 100ms Verifier 응답시간
감사로그 무결성 100% 해시 검증 일치율
Atomic Settlement 성공률 ≥ 99% DvP 완결 케이스

인수 테스트 시나리오

  1. 정상 결제: Shield → 전송 → Unshield → Atomic Settlement 완결
  2. Freeze 케이스: Blacklist 계정 → 거래 차단 확인
  3. 감사 복호: 2-of-3 승인 → 특정 TX 복호 → 원본 데이터 확인
  4. 장애 복구: 노드 1개 다운 시 서비스 연속성
